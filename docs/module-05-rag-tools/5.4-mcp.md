In the last section, we learned that AI models use **Tools** to access the real world. But imagine if every different brand of phone used a different type of charger. It would be a nightmare! 

For a long time, AI was exactly like that. Every model (ChatGPT, Claude, Gemini) had its own way of connecting to tools. To fix this, a new open standard was created called the **[Model Context Protocol (MCP)](../glossary.md#model-context-protocol)**.

## What is MCP?

MCP is like a "Universal Serial Bus (USB)" for AI. It provides a standard way for an AI model (the **Client**) to talk to an external data source or tool (the **Server**).

With MCP:
1.  **Developers** only have to write their tool once.
2.  **AI Models** can instantly connect to any MCP-compliant tool in the world.

## How the MCP Architecture Works

The MCP system has three main parts:

1.  **The MCP Server:** This is where the actual code lives. It could be a server that searches GitHub, reads a Google Sheet, or queries a SQLite database. 
2.  **The MCP Client:** This is the AI application (like Claude Desktop or a custom Python script). It "talks" to the server and asks for data.
3.  **The LLM:** The model itself, which receives the data from the client and uses it to answer your questions.

## Examples of MCP Tools

There are already hundreds of pre-built MCP servers that you can plug into your AI:
*   **File System:** Allows the AI to safely read and write files on your computer.
*   **GitHub:** Let's the AI search repositories, read issues, and even create pull requests.
*   **Slack/Discord:** Allows the AI to read messages and post updates to your team.
*   **PostgreSQL/SQLite:** Gives the AI a direct connection to your databases.

## Why MCP is the Future

MCP is the final piece of the puzzle for building **Autonomous [Agents](../glossary.md#agent)**. 

By standardizing how an AI accesses context ([RAG](5.2-rag.md)) and actions (Tools), we are moving away from simple chatbots and toward "Multi-Agent Systems" where many specialized AI workers can collaborate to solve complex problems.

## MCP Architecture in Action

To see how MCP connects clients to servers, here's a simple Python example:

```python
import json

# Simulate an MCP Server that provides tools
class MCPServer:
    def __init__(self):
        self.tools = {
            "file_system": {"name": "file_system", "description": "Read/write files on disk"},
            "github": {"name": "github", "description": "Search and interact with GitHub repos"},
            "database": {"name": "database", "description": "Query SQL databases"}
        }
    
    def list_tools(self):
        return self.tools

# Simulate an MCP Client (the AI application)
class MCPClient:
    def __init__(self, server):
        self.server = server
    
    def get_available_tools(self):
        return self.server.list_tools()

# Simulate the LLM using tools
server = MCPServer()
client = MCPClient(server)

print("Available MCP Tools:")
for tool_name, tool_info in client.get_available_tools().items():
    print(f"  - {tool_info['name']}: {tool_info['description']}")
```

Running this prints:
```text
Available MCP Tools:
  - file_system: Read/write files on disk
  - github: Search and interact with GitHub repos
  - database: Query SQL databases
```

**2. How does MCP differ from RAG in terms of what information it provides?**

**RAG** retrieves *static text content* from documents (books, articles) and adds it to context. **MCP** is a *protocol* for two-way communication: the AI can request data or trigger actions, and the server responds dynamically. RAG is read-only retrieval; MCP enables bidirectional interaction with tools and services.
